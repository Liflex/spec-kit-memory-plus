## Clarifications

### Session 2025-03-11

- **Q**: Как `/speckit.implementloop` соотносится с `/speckit.implement`? → **A**: `/speckit.implementloop` объединяет implement + quality loop в одной команде (автоматический переход к оценке без ожидания команды пользователя). `/speckit.implement` остаётся без изменений.

- **Q**: Куда ещё добавляется security интеграция из ai-factory? → **A**: Также к процессу "Создание агентов: Автоматическое создание агентов на основе изученных паттернов" в Memory System.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Объединённая реализация с quality loop (Priority: P1)

Разработчик использует команду `/speckit.implementloop` для автоматической реализации задач с немедленным переходом к итеративному циклу оценки качества и исправлениями, не дожидаясь отдельной команды.

**Why this priority**: Это основная ценность фичи — один запуск команды выполняет всю работу: реализация → оценка → критика → исправление → повтор. Не нужно ждать завершения implement и запускать отдельную команду для quality loop.

**Independent Test**: Можно протестировать, запустив `/speckit.implementloop` на существующем файле tasks.md и наблюдая за последовательным выполнением: сначала реализация задач, затем автоматические итерации оценки до threshold или лимита.

**Acceptance Scenarios**:

1. **Given** разработчик выполнил `/speckit.implementloop`, **When** команда запускается, **Then** система сначала реализует все задачи из tasks.md (как в обычном `/speckit.implement`)
2. **Given** реализация завершена, **When** все задачи выполнены, **Then** система автоматически переходит к quality loop без ожидания команды пользователя
3. **Given** quality loop запущен, **When** оценка начинается, **Then** система оценивает результат по явным правилам с вычислением score
4. **Given** оценка не прошла threshold, **When** score < 0.8 (Phase A), **Then** система автоматически генерирует критику провалившихся правил и исправляет артефакт
5. **Given** автоматическая коррекция применена, **When** исправление завершено, **Then** система повторно оценивает результат (Phase A → B при достижении threshold A)
6. **Given** цикл повторяется, **When** достигнут threshold B (score ≥ 0.9) или лимит итераций (по умолчанию 4), **Then** цикл завершается с финальным отчётом
7. **Given** stagnation detected, **When** score улучшается менее чем на 0.02 за 2 итерации, **Then** цикл останавливается с причиной "stagnation"

### User Story 2 - Рекомендация quality loop после обычной реализации (Priority: P2)

Разработчик использует обычную команду `/speckit.implement` для выполнения задач. После завершения система показывает рекомендацию о возможности запуска отдельного quality loop (`/speckit.loop`) с объяснением пользы и примером аргументов.

**Why this priority**: Это менее критично, чем автоматический цикл, но обеспечивает плавное внедрение и обучение пользователей. Рекомендация подталкивает к использованию quality loop без принуждения.

**Independent Test**: Можно протестировать, выполнив `/speckit.implement` на любом tasks.md и проверив, что в конце вывода появляется чёткая рекомендация с описанием пользы, примером аргументов для `/speckit.loop` и командой для запуска.

**Acceptance Scenarios**:

1. **Given** разработчик выполнил `/speckit.implement`, **When** все задачи завершены, **Then** система показывает секцию рекомендации "Quality Loop Available"
2. **Given** рекомендация показана, **When** пользователь читает её, **Then** видит описание пользы (score-based оценка, автоматические исправления, повышение качества кода)
3. **Given** рекомендация показана, **When** пользователь видит пример аргументов, **Then** показан пример: `/speckit.loop --criteria code-gen --max-iterations 4`
4. **Given** рекомендация показана, **When** пользователь видит объяснение, **Then** описано зачем нужны аргументы (criteria template, max iterations) и какие значения доступны
5. **Given** пользователь копирует команду, **When** запускает `/speckit.loop` с показанными аргументами, **Then** система использует уже реализованный код как отправную точку для quality loop
6. **Given** пользователь запускает `/speckit.loop` без аргументов, **When** команда выполняется, **Then** используются значения по умолчанию (criteria: auto-detect, max-iterations: 4)

### User Story 3 - Отдельный quality loop на реализованном коде (Priority: P1)

Разработчик запускает отдельную команду `/speckit.loop` на уже реализованном коде для запуска итеративного цикла оценки качества и исправлений без повторной реализации задач.

**Why this priority**: Это важно для случаев, когда код уже реализован (через `/speckit.implement` или вручную) и нужно только quality loop. Это также позволяет использовать quality loop независимо от implement.

**Independent Test**: Можно протестировать, выполнив `/speckit.implement` для реализации задач, затем `/speckit.loop` для запуска quality loop на том же коде, и наблюдая за итерациями без повторной реализации.

**Acceptance Scenarios**:

1. **Given** код уже реализован (через `/speckit.implement` или вручную), **When** разработчик выполняет `/speckit.loop`, **Then** система НЕ повторяет реализацию задач, а сразу запускает quality loop
2. **Given** `/speckit.loop` запущен, **When** оценка начинается, **Then** система оценивает существующий код по явным правилам с вычислением score
3. **Given** оценка не прошла threshold, **When** score < 0.8, **Then** система генерирует критику и исправления
4. **Given** критика применена, **When** исправление завершено, **Then** система повторно оценивает результат
5. **Given** цикл завершён, **When** достигнут threshold или лимит, **Then** показывается финальный отчёт с изменениями
6. **Given** пользователь указал `--criteria code-gen`, **When** quality loop запущен, **Then** используются правила из criteria template "code-gen"
7. **Given** пользователь указал `--max-iterations 6`, **When** quality loop запущен, **Then** максимум 6 итераций вместо стандартных 4

### User Story 4 - Безопасность SkillsMP при получении и создании скиллов (Priority: P1)

Разработчик получает скиллы из SkillsMP или создаёт новые скиллы. Система автоматически сканирует их на наличие вредоносных инструкций (prompt injection, data exfiltration, destructive commands) используя двухуровневую модель из ai-factory security.md.

**Why this priority**: Безопасность критична при получении кода извне. Вредоносный скилл может украсть учётные данные, выполнить деструктивные команды или внедрить恶意 инструкции. Это P1 потому что напрямую влияет на безопасность системы.

**Independent Test**: Можно протестировать, получив подозрительный скилл из SkillsMP с известными паттернами угроз (prompt injection, curl с .env файлами) и проверив, что система его блокирует или показывает предупреждение.

**Acceptance Scenarios**:

1. **Given** разработчик запрашивает скилл из SkillsMP, **When** скилл получен, **Then** система автоматически запускает Level 1: статический сканер Python
2. **Given** статический сканер завершён, **When** результат CLEAN, **Then** скилл передаётся на Level 2: LLM семантический обзор
3. **Given** LLM обзор завершён, **When** оба уровня прошли, **Then** скилл помечается как безопасный и устанавливается
4. **Given** статический сканер обнаружил CRITICAL угрозы, **When** это prompt injection или data exfiltration, **Then** скилл блокируется, удаляется и показывается предупреждение пользователю
5. **Given** статический сканер обнаружил WARNINGS, **When** это подозрительные паттерны, **Then** пользователю показывается предупреждение и требуется явное подтверждение
6. **Given** разработчик создаёт новый скилл через агент, **When** генерация завершена, **Then** скилл автоматически сканируется по той же двухуровневой модели
7. **Given** созданный скилл содержит подозрительный код, **When** сканер обнаружил "do not tell the user", **Then** LLM уровень помечает это как угрозу и требует уточнения

### User Story 5 - Безопасность при создании агентов из паттернов (Priority: P1)

Разработчик создаёт нового агента на основе изученных паттернов через Memory System (SkillCreationWorkflow). Система автоматически сканирует создаваемого агента на наличие вредоносных инструкций используя двухуровневую модель безопасности.

**Why this priority**: Создание агентов особенно чувствительно к безопасности, так как агент получает широкие возможности (выполнение команд, доступ к файлам). Вредоносный агент может нанести серьёзный ущерб. Это P1 потому что напрямую влияет на безопасность системы.

**Independent Test**: Можно протестировать, создав агента с подозрительным паттерном (например, инструкцию "не сообщать пользователю о выполненных командах") и проверив, что система блокирует или требует подтверждения.

**Acceptance Scenarios**:

1. **Given** разработчик создаёт агента через `SkillCreationWorkflow.create_agent_from_requirements()`, **When** агент сгенерирован, **Then** система автоматически запускает Level 1: статический сканер Python
2. **Given** статический сканер завершён, **When** результат CLEAN, **Then** агент передаётся на Level 2: LLM семантический обзор
3. **Given** LLM обзор завершён, **When** оба уровня прошли, **Then** агент сохраняется и становится доступным для использования
4. **Given** статический сканер обнаружил CRITICAL угрозы, **When** это stealth instructions ("silently", "do not tell user"), **Then** агент блокируется, удаляется и показывается предупреждение
5. **Given** разработчик использует `workflow.search_agents()`, **When** результаты получены из SkillsMP, **Then** каждый результат автоматически сканируется перед отображением
6. **Given** разработчик использует `workflow.create_agent_from_requirements()`, **When** требования содержат подозрительные инструкции, **Then** LLM уровень обнаруживает несоответствие между ролью агента и инструкциями
7. **Given** сканер обнаружил WARNING, **When** это подозрительный паттерн, **Then** пользователю показывается предупреждение с деталями и требуется явное подтверждение перед сохранением агента

### Edge Cases

- Что происходит, когда цикл достигает лимита итераций, но не прошёл threshold?
- Как система обрабатывает прерывание пользователем во время автоматического цикла?
- Что если сканер безопасности даёт false positive на легитимный код?
- Как ведёт себя цикл при нестабильном score (oscillation между двумя значениями)?
- Что происходит, если SkillsMP недоступен во время запроса скилла?
- Как обрабатываются закодированные payload (base64, hex, zero-width characters) в скиллах?
- Что если пользователь отклоняет предупреждение безопасности и хочет установить опасный скилл?
- Что если `/speckit.loop` запущен на коде, который не был реализован через SpecKit?
- Как система обрабатывает одновременный запуск `/speckit.loop` в нескольких сессиях?
- Что если паттерн для создания агента содержит легитимные команды управления системой (rm, chmod)?

## Requirements *(mandatory)*

### Functional Requirements

#### Quality Loop Commands

- **FR-001**: Система ДОЛЖНА предоставлять команду `/speckit.implementloop` для объединённой реализации задач и quality loop
- **FR-002**: Команда `/speckit.implementloop` ДОЛЖНА сначала реализовывать все задачи из tasks.md (как `/speckit.implement`), затем автоматически запускать quality loop без ожидания команды пользователя
- **FR-003**: Система ДОЛЖНА предоставлять команду `/speckit.loop` для отдельного quality loop на уже реализованном коде (без повторной реализации задач)
- **FR-004`: Команда `/speckit.loop` ДОЛЖНА поддерживать аргументы: `--criteria <template>` (api-spec, code-gen, docs, config), `--max-iterations <N>`, `--threshold-a <0.0-1.0>`, `--threshold-b <0.0-1.0>`
- **FR-005`: При запуске `/speckit.loop` без аргументов ДОЛЖНЫ использоваться значения по умолчанию: criteria auto-detect, max-iterations 4, threshold-a 0.8, threshold-b 0.9
- **FR-006**: После завершения `/speckit.implement` система ДОЛЖНА показывать рекомендацию о `/speckit.loop` с описанием пользы, примером аргументов и объяснением зачем они нужны
- **FR-007**: Рекомендация ДОЛЖНА включать: описание score-based оценки, объяснение автоматических исправлений, пример команды `/speckit.loop --criteria code-gen --max-iterations 4`, список доступных criteria templates

#### Quality Loop Evaluation

- **FR-008**: Система ДОЛЖНА поддерживать фазовую модель оценки (Phase A: threshold 0.8, Phase B: threshold 0.9)
- **FR-009**: Система ДОЛЖНА автоматически генерировать критику провалившихся правил (max 5 issues за итерацию)
- **FR-010**: Система ДОЛЖНА автоматически применять целевые исправления только для провалившихся областей
- **FR-011**: Система ДОЛЖНА повторять цикл оценки → исправления → переоценки до достижения threshold или лимита итераций (по умолчанию 4)
- **FR-012**: Система ДОЛЖНА останавливать цикл при stagnation (улучшение score < 0.02 за 2 итерации)
- **FR-013**: Система ДОЛЖНА вычислять score по формуле: `score = sum(passed_weights) / sum(all_active_weights)`
- **FR-014**: Система ДОЛЖНА поддерживать уровни severity для правил: fail (weight 2), warn (weight 1), info (weight 0)
- **FR-015**: Passed = (score >= threshold) AND (no fail-severity rules failed)
- **FR-016**: Система ДОЛЖНА предоставлять переиспользуемые criteria templates для разных типов задач (API-spec, code-gen, docs, config)
- **FR-017**: Финальный отчёт quality loop ДОЛЖЕН включать: iteration/max_iterations, phase, final_score, stop_reason, distance to success (если не пройден), список изменённых файлов

#### Security Integration - SkillsMP

- **FR-018**: При получении скиллов из SkillsMP система ДОЛЖНА автоматически запускать Level 1: статический сканер Python
- **FR-019**: Статический сканер ДОЛЖЕН обнаруживать: prompt injection patterns, data exfiltration attempts, stealth instructions, destructive commands, config tampering, encoded payloads, social engineering, suspicious HTML comments
- **FR-020**: Система ДОЛЖНА запускать Level 2: LLM семантический обзор после прохождения Level 1
- **FR-021**: LLM обзор ДОЛЖЕН оценивать: соответствие инструкций заявленной цели скилла, запросы доступа к чувствительным данным, подозрительные комбинации инструментов, попытки манипуляции через urgency/authority
- **FR-022**: Скиллы с CRITICAL угрозами ДОЛЖНЫ блокироваться и удаляться без возможности override
- **FR-023**: Скиллы с WARNINGS ДОЛЖНЫ требовать явного подтверждения от пользователя
- **FR-024**: При создании новых скиллов система ДОЛЖНА автоматически сканировать их по той же двухуровневой модели
- **FR-025**: Система ДОЛЖНА поддерживать smart code-block awareness: паттерны внутри markdown code blocks демуются до warnings (как документация)
- **FR-026`: Система ДОЛЖНА сохранять результаты сканирования безопасности в history.jsonl для аудита

#### Security Integration - Agent Creation

- **FR-027`: При создании агентов через `SkillCreationWorkflow.create_agent_from_requirements()` система ДОЛЖНА автоматически запускать Level 1: статический сканер Python
- **FR-028`: Статический сканер для агентов ДОЛЖЕН дополнительно проверять: широкие системные права (shell access, file write), комбинации инструментов с высоким риском (curl + ~/.ssh, rm -rf), инструкции обхода уведомлений ("do not tell user", "silently")
- **FR-029`: Система ДОЛЖНА запускать Level 2: LLM семантический обзор для созданных агентов
- **FR-030`: LLM обзор для агентов ДОЛЖЕН оценивать: соответствие инструкций заявленной роли агента, наличие-stealth паттернов, подозрительные системные команды, несоответствие между role/personality и фактическими инструкциями
- **FR-031**: Агенты с CRITICAL угрозами ДОЛЖНЫ блокироваться и удаляться без возможности сохранения
- **FR-032`: Агенты с WARNINGS ДОЛЖНЫ требовать явного подтверждения перед сохранением в файл
- **FR-033`: При поиске агентов через `workflow.search_agents()` результаты из SkillsMP ДОЛЖНЫ автоматически сканироваться перед отображением
- **FR-034`: При создании агента из требований, содержащих подозрительные инструкции, LLM уровень ДОЛЖЕН обнаруживать несоответствие и запрашивать подтверждение

### Key Entities

- **Quality Rule**: Правило оценки с атрибутами id, description, severity (fail/warn/info), weight (2/1/0), phase (A/B), check
- **Criteria Template**: Переиспользуемый набор правил для типа задач (API-spec, code-gen, docs, config)
- **Evaluation Result**: Результат оценки с score, passed (boolean), failed_rules list, warnings list
- **Security Threat**: Обнаруженная угроза с типом (prompt injection, data exfiltration, и т.д.), severity (CRITICAL/WARNING), деталями
- **Skill Scan Report**: Отчёт о сканировании скилла с результатами Level 1 (статический) и Level 2 (LLM)
- **Agent Scan Report**: Отчёт о сканировании агента с результатами Level 1 и Level 2, плюс агент-специфичные проверки (system access, stealth patterns)
- **Loop State**: Состояние цикла с iteration, max_iterations, phase, current_score, last_score, stagnation_count

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователи могут запустить `/speckit.implementloop` и получить реализацию с автоматическим улучшением качества за менее чем 5 минут без ручного вмешательства
- **SC-002**: 90% автоматических циклов достигают threshold B (score ≥ 0.9) или останавливаются с понятной причиной (iteration_limit, stagnation)
- **SC-003**: Пользователи видят рекомендацию `/speckit.loop` после каждого `/speckit.implement` с примером аргументов и объяснением
- **SC-004**: Команда `/speckit.loop` запускается на уже реализованном коде без повторной реализации задач
- **SC-005**: Все полученные из SkillsMP скиллы проходят двухуровневое сканирование безопасности за менее чем 30 секунд
- **SC-006**: 100% скиллов и агентов с CRITICAL угрозами блокируются; ни один опасный артефакт не устанавливается без явного подтверждения
- **SC-007**: Система предотвращает установку вредоносных скиллов и агентов с prompt injection, data exfiltration, stealth instructions, или destructive commands
- **SC-008**: Финальный отчёт quality loop включает distance to success (numeric gap до threshold) для незавершённых циклов
- **SC-009**: Criteria templates позволяют быстрый старт для типовых задач (API-spec, code-gen, docs, config) без определения правил с нуля
- **SC-010**: Все созданные агенты автоматически сканируются по двухуровневой модели безопасности перед сохранением

---

## Assumptions

1. Пользователь знаком с базовыми командами SpecKit (`/speckit.specify`, `/speckit.plan`, `/speckit.tasks`, `/speckit.implement`)
2. Проект имеет структуру SpecKit с директорией `specs/` и файлами `spec.md`, `plan.md`, `tasks.md`
3. Python доступен в системе для запуска статического сканера безопасности
4. LLM (Claude Code или другой AI-ассистент) доступен для семантического обзора скиллов и агентов
5. SkillsMP API доступен для поиска и получения скиллов
6. Memory System с `SkillCreationWorkflow` доступна для создания агентов
7. По умолчанию используется 4 итерации цикла (может быть настроено через `--max-iterations`)
8. Пороги по умолчанию: Phase A = 0.8, Phase B = 0.9 (могут быть переопределены через аргументы)
9. Stagnation threshold = 0.02 (изменение score за 2 итерации)
10. Файловая структура для loop state: `.speckit/evolution/` аналогично ai-factory
11. Память проекта (`~/.claude/memory/`) доступна для сохранения уроков о безопасности
12. `/speckit.implement` НЕ изменяется в логике выполнения задач, только добавляется рекомендация в конце
13. `/speckit.loop` может быть запущен на любом коде, не только реализованном через SpecKit

---

## Out of Scope

1. Визуальный интерфейс для просмотра истории циклов (только текстовый вывод в терминал)
2. Интеграция с CI/CD системами для автоматического запуска quality loops
3. Распределённое выполнение циклов (только локальное выполнение)
4. Графические отображения score progression (только текстовые таблицы)
5. Автоматическое создание PR в git после успешного цикла
6. Интеграция с внешними системами отслеживания багов (Jira, GitHub Issues)
7. Обучение системы на предыдущих циклах для автоматического улучшения criteria templates
8. Изменение логики `/speckit.implement` (только добавление рекомендации в конце)
9. Автоматическое исправление CRITICAL угроз в скиллах/агентах (только блокировка)
10. Создание собственных правил безопасности (используются правила из ai-factory security.md)
