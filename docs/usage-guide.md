## Обзор: две системы в одном

Spec Kit состоит из двух связанных, но независимых систем:

### Система 1: Spec-Driven Development (разработка через спецификации)

Полный цикл от идеи до кода:

```
Идея --> Спецификация --> Уточнение --> План --> Задачи --> Код
         /specify        /clarify      /plan    /tasks    /implement
```

**Когда использовать:** вы начинаете новую фичу или проект и хотите, чтобы ИИ-агент реализовал её качественно, а не "как получится".

### Система 2: Quality Management (управление качеством)

Оценка и улучшение уже написанного кода:

```
Код --> Оценка --> Критика --> Улучшение --> Повтор
       /loop      (автоматически внутри loop)
```

**Когда использовать:** у вас уже есть код, и вы хотите проверить/улучшить его качество по набору правил.

### Как они связаны

Можно использовать каждую систему отдельно. Но максимальная ценность — вместе:

```
/specify --> /clarify --> /plan --> /tasks --> /implementloop
                                                    |
                                              implement + loop
```

Команда `/speckit.implementloop` объединяет обе системы: сначала реализует задачи, затем автоматически запускает quality loop на результате.

## Быстрый выбор: какую команду использовать?

### "Я хочу..."

| Ситуация | Команда | Пояснение |
|----------|---------|-----------|
| Начать новый проект с нуля | `/speckit.constitution` | Задайте правила и принципы проекта |
| Описать новую фичу | `/speckit.specify` | Создаст spec.md с требованиями |
| Уточнить размытые требования | `/speckit.clarify` | Задаст вопросы и дополнит spec.md |
| Создать технический план | `/speckit.plan` | Создаст plan.md с архитектурой |
| Разбить план на задачи | `/speckit.tasks` | Создаст tasks.md с порядком выполнения |
| Проверить что spec/plan/tasks согласованы | `/speckit.analyze` | Аудит без изменения файлов |
| Реализовать фичу | `/speckit.implement` | Выполнит все задачи из tasks.md |
| Реализовать + проверить качество | `/speckit.implementloop` | implement + quality loop |
| Сделать мелкий фикс (< 4 часов) | `/speckit.features` | Минимальный spec+plan+tasks за раз |
| Проверить качество кода | `/speckit.loop` | Итеративная оценка по правилам |
| Настроить CI/CD quality gate | `/speckit.loop --gate-preset` | Политики прохождения через флаг |
| Управлять планами качества | `/speckit.plans` | Планы улучшения качества |
| Создать GitHub Issues из задач | `/speckit.taskstoissues` | Экспорт tasks.md в Issues |

### Размер задачи --> Подход

| Размер | Пример | Подход |
|--------|--------|--------|
| **Мелкий фикс** (минуты) | Исправить опечатку, добавить поле | Сделать вручную |
| **Маленькая фича** (< 4 часов) | Добавить валидацию, новый endpoint | `/speckit.features` |
| **Средняя фича** (1-3 дня) | Новый модуль, интеграция с API | `/specify` -> `/clarify` -> `/plan` -> `/tasks` -> `/implement` |
| **Большая фича** (неделя+) | Новая подсистема, рефакторинг архитектуры | Полный цикл + разбивка на фазы |

## Рабочий процесс разработки

### 1. `/speckit.constitution` --- конституция проекта

**Зачем:** задать фундаментальные правила проекта, которые будут влиять на все последующие спецификации и планы.

**Когда использовать:**
- В начале нового проекта (один раз)
- При смене архитектурного подхода
- Когда нужно зафиксировать принципы для команды

**Пример:**

```
/speckit.constitution Проект следует принципу "Library-First" — каждая фича сначала реализуется как библиотека. Используем строгий TDD. Предпочитаем функциональные паттерны. Все API должны быть REST-first. Минимум зависимостей.
```

**Что создаёт:** файл `constitution.md` в директории спецификации, который подгружается во все остальные команды `/speckit.*`.

**Не нужно если:** вы работаете над существующим проектом с уже сложившимися практиками и не хотите их формализовать.

---

### 2. `/speckit.specify` --- спецификация фичи

**Зачем:** превратить идею в структурированное описание требований, которое понятно и человеку, и ИИ-агенту.

**Когда использовать:**
- Начинаете новую фичу
- Хотите зафиксировать что именно нужно сделать, прежде чем писать код
- Работаете в команде и нужен общий документ

**Пример:**

```
/speckit.specify Создать систему уведомлений для пользователей. Пользователь должен получать уведомления о новых комментариях к его задачам. Уведомления отображаются в выпадающем меню в шапке. Непрочитанные выделяются. Можно отметить все как прочитанные.
```

**Что создаёт:** файл `spec.md` с структурированными требованиями: цели, пользовательские сценарии, ограничения, критерии приёмки.

**Важно:** описывайте ЧТО и ЗАЧЕМ, а не КАК. Технические детали — на этапе `/plan`.

---

### 3. `/speckit.clarify` --- уточнение спецификации

**Зачем:** найти и устранить пробелы в спецификации до написания кода. ИИ задаёт до 5 целевых вопросов и записывает ответы.

**Когда использовать:**
- После `/specify`, чтобы доуточнить детали
- Когда спецификация написана другим человеком и содержит неясности
- Когда вы сами не уверены в некоторых требованиях

**Пример — общее уточнение:**

```
/speckit.clarify
```

ИИ сам найдёт слабые места и задаст вопросы.

**Пример — фокусное уточнение:**

```
/speckit.clarify Уточни требования к безопасности и производительности. Сколько уведомлений может быть у пользователя? Как долго хранить старые?
```

**Что делает:** обновляет `spec.md`, добавляя ответы на вопросы в соответствующие секции.

**Можно вызывать несколько раз** — каждый вызов углубляет спецификацию.

---

### 4. `/speckit.plan` --- технический план

**Зачем:** превратить требования (spec.md) в технический план реализации: архитектура, технологии, структура файлов, API-контракты.

**Когда использовать:**
- После завершения спецификации (`/specify` + `/clarify`)
- Когда нужно согласовать техническое решение до написания кода

**Пример:**

```
/speckit.plan Используем PostgreSQL для хранения уведомлений. Backend на FastAPI с WebSocket для real-time доставки. Frontend — React-компонент NotificationBell. Очередь через Redis pub/sub.
```

**Что создаёт:** файл `plan.md` с архитектурой, структурой файлов, API-контрактами, диаграммами зависимостей.

**Важно:** здесь вы указываете технические предпочтения. Если не укажете — ИИ выберет сам на основе spec.md и конституции.

---

### 5. `/speckit.tasks` --- генерация задач

**Зачем:** разбить план на конкретные, упорядоченные задачи с зависимостями, которые можно выполнять последовательно.

**Когда использовать:**
- После `/plan`, перед началом реализации
- Когда хотите увидеть объём работы до начала кодирования

**Пример:**

```
/speckit.tasks
```

Аргументы обычно не нужны — команда работает на основе spec.md и plan.md.

**Что создаёт:** файл `tasks.md` со списком задач, каждая содержит:
- Название и описание
- Зависимости от других задач
- Конкретные файлы для создания/изменения
- Критерии завершения

---

### 6. `/speckit.analyze` --- аудит артефактов

**Зачем:** проверить что spec.md, plan.md и tasks.md согласованы между собой. Не изменяет файлы — только анализирует.

**Когда использовать:**
- После `/tasks`, перед `/implement` — как финальная проверка
- Когда вносили ручные правки в spec/plan/tasks и хотите убедиться в согласованности

**Пример:**

```
/speckit.analyze
```

**Что проверяет:**
- Все требования из spec.md покрыты задачами
- Зависимости в tasks.md не конфликтуют
- Технические решения в plan.md соответствуют spec.md
- Нет "висящих" задач без привязки к требованиям

---

### 7. `/speckit.implement` --- реализация

**Зачем:** автоматически выполнить все задачи из tasks.md: создать файлы, написать код, тесты.

**Когда использовать:**
- После полного цикла specify -> clarify -> plan -> tasks -> analyze
- Когда tasks.md готов и проверен

**Пример:**

```
/speckit.implement
```

**Как работает:**
1. Читает tasks.md
2. Выполняет задачи в порядке зависимостей
3. Для каждой задачи: создаёт/изменяет файлы, пишет тесты
4. Сохраняет знания в память агента

**Важно:** для крупных проектов рекомендуется реализовывать фазами — сначала ядро, потом расширения.

---

## Быстрая разработка

### `/speckit.features` --- быстрые фичи

**Зачем:** для мелких задач (< 4 часов), где полный цикл specify->plan->tasks избыточен. Создаёт минимальный spec, plan и tasks за один вызов.

**Когда использовать:**
- Небольшое улучшение или фикс
- Добавление одного endpoint, компонента, утилиты
- Когда не хочется проходить весь 6-шаговый процесс

**Пример:**

```
/speckit.features Добавить endpoint GET /api/users/:id/stats, который возвращает количество задач пользователя по статусам. Использовать существующую модель Task из models/task.py.
```

**Что делает:** сразу генерирует spec.md, plan.md, tasks.md с минимальным наполнением.

---

### `/speckit.checklist` --- чеклист

**Зачем:** создать кастомный чеклист для текущей фичи на основе ваших требований.

**Когда использовать:**
- Перед code review — убедиться что всё сделано
- Для ручной проверки — когда автоматический loop не нужен
- Для команды — как список требований для ревью

**Пример:**

```
/speckit.checklist Проверить: безопасность API, покрытие тестами > 80%, документация endpoints, обработка ошибок, валидация входных данных.
```

## Контроль качества

### `/speckit.loop` --- цикл оценки качества

**Зачем:** итеративно проверять и улучшать код по набору правил. Цикл: оценка -> скоринг -> критика -> исправление -> повтор.

**Когда использовать:**
- После реализации фичи — проверить качество
- Перед code review — автоматически поднять качество
- В CI/CD — как quality gate
- Для ревью legacy кода — найти проблемы

**Базовые примеры:**

```bash
# Оценить backend-код
/speckit.loop --criteria backend

# Оценить по нескольким шаблонам
/speckit.loop --criteria backend,security,testing

# Автоматически подобрать шаблоны по типу проекта
/speckit.loop --project-type web-app

# Использовать готовый пресет (комбинация шаблонов)
/speckit.loop --blend-preset full_stack_secure
```

**Управление строгостью:**

```bash
# Строгая проверка (production-ready)
/speckit.loop --gate-preset production --strict

# Мягкая проверка (для development)
/speckit.loop --criteria backend --gate-preset development

# Больше итераций для сложного кода
/speckit.loop --criteria security --max-iterations 6
```

**Вывод результатов:**

```bash
# HTML-отчёт
/speckit.loop --html-output report.html

# JSON для CI/CD
/speckit.loop --json-output report.json

# Карточка результатов в консоли
/speckit.loop --show-result-card
```

**Как работает внутри:**

```
Iteration 1/4 | Phase A | Score: 0.72 | FAIL
  Failed: correctness.tests, quality.error_handling
  --> Critique: "Отсутствуют тесты для edge cases, нет обработки null..."
  --> Refiner: применяет исправления

Iteration 2/4 | Phase A | Score: 0.84 | PASS --> Phase B
  --> Переход к более строгим правилам

Iteration 3/4 | Phase B | Score: 0.92 | PASS
  Stop Reason: threshold_reached
```

**13 встроенных шаблонов:** backend, frontend, security, performance, testing, api-spec, database, docs, config, infrastructure, ui-ux, live-test, code-gen.

Подробнее о шаблонах: [Quality Loop](quality-loop.md).

---

### `/speckit.implementloop` --- реализация + качество

**Зачем:** объединяет `/speckit.implement` и `/speckit.loop` в одну команду.

**Когда использовать:**
- Когда хотите сразу после реализации проверить качество кода
- Основной рабочий режим для spec-driven development

**Пример:**

```bash
/speckit.implementloop --criteria code-gen --max-iterations 4
```

**Как работает:**
1. **Фаза 1:** выполняет все задачи из tasks.md (как `/implement`)
2. **Фаза 2:** запускает quality loop на изменённых файлах (как `/loop`)

## Управление качеством

Основные инструменты управления качеством --- это `/speckit.loop` (итеративная оценка и улучшение) и `/speckit.templates` (выбор критериев). Для планирования улучшений используйте `/speckit.plans`.

### `/speckit.plans` --- планы качества

**Зачем:** создавать и отслеживать планы улучшения качества проекта.

**Когда использовать:**
- Хотите формализовать шаги по улучшению качества
- Нужен структурированный подход к техническому долгу
- Готовите план рефакторинга или улучшений для команды

**Примеры:**

```bash
# Создать план улучшения
/speckit.plans create

# Просмотреть текущий план
/speckit.plans show

# Обновить статус
/speckit.plans update
```

## Шаблоны

### `/speckit.templates` --- реестр шаблонов

**Зачем:** управление шаблонами критериев качества — просмотр, поиск, сравнение, комбинирование.

**Когда использовать:**
- Не знаете какой шаблон выбрать для `/loop`
- Хотите создать комбинированный шаблон из нескольких
- Сравниваете что проверяет backend vs frontend

**Примеры:**

```bash
# Список всех шаблонов
/speckit.templates list

# Подробности конкретного шаблона
/speckit.templates info backend

# Поиск шаблонов по теме
/speckit.templates search security

# Рекомендация для типа проекта
/speckit.templates recommend web-app

# Сравнить шаблоны
/speckit.templates compare frontend backend

# Объединить шаблоны
/speckit.templates blend frontend backend --mode union

# Готовые пресеты (комбинации)
/speckit.templates presets list
/speckit.templates presets info full_stack_secure
```

## Интеграции

### `/speckit.taskstoissues` --- задачи в GitHub Issues

**Зачем:** экспортировать задачи из tasks.md как GitHub Issues для трекинга в команде.

**Когда использовать:**
- Работаете в команде и используете GitHub Issues для планирования
- Хотите отслеживать прогресс реализации через Issues/Projects

**Пример:**

```
/speckit.taskstoissues
```

---

### `/speckit.tobeads` --- задачи в Beads

**Зачем:** импортировать задачи из tasks.md в Beads issue tracker с эпиками и зависимостями.

**Когда использовать:**
- Используете Beads для управления проектом

**Пример:**

```
/speckit.tobeads
```

## Полные сценарии использования

### Сценарий 1: Новая фича с нуля (средний размер)

```bash
# 1. Описать что хотим
/speckit.specify Добавить систему рейтингов для товаров. Пользователи могут ставить оценку 1-5 звёзд и оставлять текстовый отзыв. Показывать средний рейтинг на карточке товара. Модерация отзывов через админку.

# 2. Уточнить детали
/speckit.clarify Уточни: может ли пользователь изменить свой отзыв? Нужна ли привязка к покупке? Как сортировать отзывы?

# 3. Создать техплан
/speckit.plan Используем PostgreSQL, таблицы reviews и ratings. REST API на FastAPI. Frontend-компонент StarRating на React.

# 4. Создать задачи
/speckit.tasks

# 5. Проверить согласованность
/speckit.analyze

# 6. Реализовать + проверить качество
/speckit.implementloop --criteria code-gen,backend,security
```

### Сценарий 2: Мелкое улучшение

```bash
# Одна команда — всё сразу
/speckit.features Добавить пагинацию к GET /api/products. Параметры: page (default 1), limit (default 20, max 100). Вернуть total_count в ответе.
```

### Сценарий 3: Проверка качества существующего кода

```bash
# Запустить оценку backend с отчётом
/speckit.loop --criteria backend,security --html-output quality-report.html --show-result-card
```

### Сценарий 4: Настройка CI/CD quality gate

```bash
# Запустить loop с gate-preset и JSON-выводом для CI/CD
/speckit.loop --criteria backend,security --gate-preset production --json-output report.json

# В CI/CD — exit code != 0 если не прошёл gate
```

### Сценарий 5: Начало нового проекта

```bash
# 1. Определить принципы
/speckit.constitution Security-first. TDD обязателен. REST API. PostgreSQL. Docker для деплоя.

# 2. Определить первую фичу и пройти полный цикл
/speckit.specify ...
/speckit.clarify
/speckit.plan ...
/speckit.tasks
/speckit.implement
```
