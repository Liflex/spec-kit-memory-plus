## Проблема

AI-агенты для кодинга не имеют памяти. Каждая сессия начинается с нуля — никаких воспоминаний о прошлых багах, никакого знания проверенных паттернов, никакого понимания архитектурных решений. Агент совершает одни и те же ошибки, задаёт одни и те же вопросы и заново открывает одни и те же решения. Снова и снова.

## Решение

Spec Kit превращает AI-агента в обучающуюся систему. Каждая команда в spec-driven workflow — от написания спецификаций до реализации кода — читает из и пишет в персистентный слой памяти. Агент накапливает знания о проекте между сессиями: извлечённые уроки, переиспользуемые паттерны, архитектурные решения. Знания сохраняются немедленно при обнаружении, а не в конце сессии, которая может никогда не завершиться.

**Память — это не фича, прикрученная к workflow. Память *и есть* workflow.**

---

## Как это работает

```
Сессия 1: /speckit.specify → обнаруживает доменные правила → сохраняет в память
Сессия 2: /speckit.plan    → читает прошлые уроки → избегает известных ловушек
Сессия 3: /speckit.tasks   → вспоминает паттерны → генерирует лучшую декомпозицию
Сессия 4: /speckit.implement → применяет архитектурные знания → консистентный код
Сессия 5: исправление бага  → учится на ошибке → больше не повторяет
```

Каждая команда следует одному циклу:

1. **Чтение** памяти в начале (уроки, паттерны, архитектурные решения)
2. **Выполнение** задачи с накопленным контекстом
3. **Запись** инсайтов немедленно при обнаружении — не ждать завершения

## Архитектура памяти

Три уровня, каждый для своей цели:

| Уровень | Хранилище | Назначение | Накладные расходы |
|---------|-----------|-----------|-------------------|
| **Сессионная** | Нативный контекст | Состояние текущего разговора | Нет |
| **Файловая** | `.claude/memory/*.md` | Персистентные знания проекта | ~1-2% токенов |
| **Векторная** | Ollama embeddings | Семантический поиск между сессиями | Опционально |

### Файловая память (всегда доступна)

Три markdown-файла, автоматически создаются при отсутствии:

| Файл | Содержит | Записывается когда |
|------|----------|-------------------|
| `lessons.md` | Исправленные баги, обнаруженные ловушки, неочевидные правила | Агент сталкивается с нетривиальной проблемой |
| `patterns.md` | Переиспользуемые решения, проверенные подходы | Паттерн успешно применён 2+ раз |
| `architecture.md` | Технические решения с обоснованием | Приняты значимые дизайн-решения |

Перед записью агент сканирует существующие заголовки для предотвращения дубликатов. Знания чисто накапливаются со временем.

### Векторная память (опционально, требует Ollama)

Семантический поиск через CLI `vector_memory.py`. Находит релевантный прошлый опыт, даже когда ключевые слова не совпадают. Полезна для больших проектов с сотнями записей в памяти. Корректно отключается при отсутствии Ollama — без проверок, без вопросов, без ошибок.

## Команды

### Основной workflow

Полный пайплайн spec-driven разработки. Каждая команда читает память, выполняет задачу и сохраняет новые инсайты инлайново.

| Команда | Назначение |
|---------|-----------|
| `/speckit.specify` | Создание спецификации фичи из естественного языка |
| `/speckit.clarify` | До 5 целевых вопросов для уточнения неопределённостей в спеке |
| `/speckit.plan` | Генерация архитектуры и плана реализации |
| `/speckit.tasks` | Декомпозиция плана в упорядоченные по зависимостям задачи |
| `/speckit.analyze` | Кросс-артефактный анализ консистенции (только чтение) |
| `/speckit.implement` | Выполнение всех задач из tasks.md |
| `/speckit.checklist` | Генерация чеклиста качества требований |
| `/speckit.constitution` | Определение принципов и ограничений проекта |

### Быстрые и комбинированные

| Команда | Назначение |
|---------|-----------|
| `/speckit.features` | Быстрая фича для мелких задач (< 4 часов) — spec + plan + implement |
| `/speckit.loop` | Quality loop на существующем коде — итеративная оценка/критика/исправление |
| `/speckit.implementloop` | Реализация задач + quality loop в одной команде |

### Интеграции

| Команда | Назначение |
|---------|-----------|
| `/speckit.tobeads` | Импорт задач в Beads issue tracker |
| `/speckit.taskstoissues` | Конвертация задач в GitHub issues |

## Быстрый старт

### Быстрая фича (< 4 часов)

```bash
/speckit.features Исправить таймаут логина на мобильных
```

Агент исследует кодовую базу, создаст минимальный план, реализует исправление, предложит тесты и сохранит полученные знания в память.

### Полный workflow

```bash
# 1. Написать спецификацию
/speckit.specify Система аутентификации пользователей с OAuth2

# 2. Уточнить неопределённости (опционально, но рекомендуется)
/speckit.clarify

# 3. Создать архитектурный план
/speckit.plan Next.js с PostgreSQL

# 4. Сгенерировать задачи
/speckit.tasks

# 5. Проанализировать консистенцию перед реализацией (опционально)
/speckit.analyze

# 6. Реализовать
/speckit.implement
```

### Quality Loop

```bash
# Итеративное улучшение качества существующего кода
/speckit.loop --criteria code-gen

# Ревью безопасности
/speckit.loop --criteria security --max-iterations 6

# Комбинирование критериев
/speckit.loop --criteria backend,live-test

# Реализация + quality loop за один шаг
/speckit.implementloop --criteria code-gen --max-iterations 4
```

### Критерии Quality Loop (13 встроенных)

| Критерий | Применение |
|----------|-----------|
| `api-spec` | API спецификации, OpenAPI |
| `code-gen` | Генерация кода, функции, классы |
| `docs` | Документация, README, гайды |
| `config` | Конфигурационные файлы, YAML/JSON |
| `database` | Схемы БД, SQL, миграции |
| `frontend` | Frontend код, React/Vue/Angular |
| `backend` | Backend сервисы, API |
| `infrastructure` | DevOps, Docker, Kubernetes |
| `testing` | Тестовые файлы, unit/integration |
| `security` | Безопасность, auth, XSS/SQLi |
| `performance` | Оптимизация производительности |
| `ui-ux` | UI/UX дизайн, accessibility |
| `live-test` | Реальные HTTP запросы, автоматизация браузера, проверка БД |

Комбинирование через запятую: `--criteria backend,live-test`

## Установка

### Через AI-ассистента

Скажите вашему AI-ассистенту:

```
Выполни инструкции по установке из INSTALL.md
```

Ассистент:
- Скопирует шаблоны и команды Spec Kit
- Настроит ваш редактор (Claude Code или Cursor)
- Создаст структуру директорий памяти
- Опционально настроит Ollama для векторного поиска

Подробности в [INSTALL.md](../../INSTALL.md).

### Поддерживаемые редакторы

| Редактор | Уровень поддержки |
|----------|------------------|
| **Claude Code** | Полная: slash-команды, память, векторный поиск |
| **Cursor** | Через правила: `.cursor/rules/`, память, векторный поиск через терминал |

## Память на практике

### Что сохраняется

Реальные примеры записей памяти, которые создаёт агент:

**lessons.md** — после исправления бага:
```markdown
## PostgreSQL JSONB Index Ignored on Nested Queries

**Date:** 2026-03-10
**Problem:** GIN index на metadata не используется при запросах вложенных ключей с оператором @> внутри подзапроса
**Solution:** Переписать как JOIN с lateral unnest — индекс теперь используется, время запроса упало с 4.2с до 12мс
**Tags:** #postgresql #performance #jsonb
```

**patterns.md** — после обнаружения переиспользуемого подхода:
```markdown
## Optimistic Locking with Version Column

**When to use:** Любая сущность с конкурентным доступом на запись от нескольких пользователей/сервисов
**How to implement:** Добавить колонку `version INT DEFAULT 0`, включить `WHERE version = :expected` в UPDATE, обработать StaleObjectError с retry
```

### Как память протекает через workflow

```
/speckit.specify
  READ:  lessons.md — "В прошлом auth спеке пропустили OAuth refresh token flow"
  WRITE: patterns.md — "Доменное правило: все API токены должны иметь настраиваемый TTL"

/speckit.plan
  READ:  architecture.md — "Проект использует event sourcing для аудита"
  READ:  patterns.md — "Паттерн оптимистичной блокировки для конкурентных записей"
  WRITE: architecture.md — "Выбран Redis для сессий вместо JWT"

/speckit.tasks
  READ:  Все три файла для полного контекста
  WRITE: patterns.md — "Декомпозиция: всегда отделять миграцию от seed данных"
  WRITE: lessons.md — "Неочевидно: auth middleware должен быть до rate limiter"

/speckit.implement
  READ:  Все три файла перед каждой фазой
  WRITE: Отложенная запись после завершения (намеренно — избегает шума во время реализации)
```
